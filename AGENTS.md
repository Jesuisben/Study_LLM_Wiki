# LLM 위키 운영 규칙

이 Obsidian Vault는 사용자의 개발자 학습 기록을 장기적으로 축적하기 위한 **LLM 위키**다. 이 파일은 앞으로 이 Vault에서 작업하는 모든 LLM Agent가 먼저 읽어야 하는 스키마이자 운영 규칙이다.

## 1. 위키의 주인, 목적, 범위

### 주인
- 사용자는 어문계열 4년제 대학을 졸업했고, 현재 개발자가 되기 위해 국비지원 학원에서 웹서비스 풀스택 개발 과정을 공부하고 있다.
- 사용자는 학원 수업 내용을 직접 마크다운으로 정리해 `raw/` 폴더에 넣는다.

### 목적
- 사용자가 학원에서 배운 내용을 시간순으로 추적한다.
- 나중에 “며칠에 무엇을 배웠는지”, “이 개념을 전에 어떻게 설명했는지”, “지금 모르는 내용이 기존 학습 내용과 어떻게 연결되는지”를 빠르게 꺼내 쓴다.
- 사용자의 공부가 쌓일수록 LLM도 `wiki/`를 통해 함께 똑똑해지도록 한다.
- 새 채팅/새 세션에서도 과거 대화를 길게 다시 붙여넣지 않고, 이 Vault의 `AGENTS.md`, `wiki/index.md`, `wiki/log.md`, 관련 위키 페이지를 읽어 맥락을 복원한다.

### 주로 다루는 내용
국비지원 커리큘럼과 사용자의 학습 기록을 중심으로 한다.

1. Java
2. Oracle / SQL / DB 모델링
3. HTML, CSS, JavaScript, Bootstrap
4. Spring, Spring Boot
5. React + TypeScript
6. Frontend / Backend 웹서비스 구현
7. Linux
8. AWS
9. CI/CD
10. Passwordless / 인증
11. Python
12. 위 내용과 연결되는 개발 도구, 오류 해결 기록, 프로젝트 회고, 개념 비교, 면접 대비 지식

## 2. 폴더 구조

```text
./
├── AGENTS.md                 # 이 파일: LLM 위키 운영 규칙
├── raw/                      # 사용자가 넣는 원본 자료. LLM은 절대 수정하지 않는다.
│   └── assets/               # 원본 자료에 딸린 이미지/첨부파일
└── wiki/                     # LLM이 생성·수정하는 지식 베이스
    ├── index.md              # 전체 위키 목차: 모든 위키 페이지의 카탈로그
    ├── log.md                # 작업 일지: ingest/query/lint/update 기록
    ├── summaries/            # 수업/자료 단위 요약 페이지
    ├── concepts/             # 개념 페이지: Java 문법, SQL 조인, REST API 등
    ├── entities/             # 엔티티 페이지: 기술, 도구, 프레임워크, 라이브러리 등
    ├── comparisons/          # 비교 페이지: JSP vs Thymeleaf, Spring MVC vs REST 등
    ├── queries/              # 사용자의 질문에 대한 보존 가치 있는 답변
    └── _meta/                # 위키 운영 보조 문서, lint 리포트, 토픽맵 등
```

### `raw/` 규칙
- `raw/`는 원본 자료 레이어다. **LLM은 `raw/` 안의 파일을 수정, 삭제, 재작성하지 않는다.**
- 원본 자료의 오탈자나 잘못된 설명을 발견해도 `raw/`를 고치지 않는다. 대신 `wiki/` 페이지에서 “원본에는 X라고 되어 있으나, 현재 정리는 Y”처럼 기록한다.
- 사용자가 수업 노트, 실습 기록, 캡처, 참고 링크를 `raw/`에 추가하면 LLM은 읽어서 `wiki/`에 통합한다.
- 이미지가 있으면 `raw/assets/`에 두고, 필요할 때 별도로 확인한 뒤 위키 페이지에서 `![[...]]` 또는 파일 경로로 참조한다.

### `wiki/` 규칙
- `wiki/`는 LLM이 유지보수하는 컴파일된 지식 레이어다.
- 모든 페이지는 Markdown이며, 노트끼리 연결할 때 Obsidian 위키링크 `[[ ]]`를 사용한다.
- 모든 새 페이지는 `wiki/index.md`에 등록하고, 모든 의미 있는 작업은 `wiki/log.md`에 기록한다.

## 3. 세션 시작 규칙

이 Vault에서 작업을 시작하면, 어떤 작업이든 먼저 다음을 읽고 방향을 잡는다.

1. `AGENTS.md` — 운영 규칙과 사용자의 목적 확인
2. `wiki/index.md` — 어떤 페이지가 이미 있는지 확인
3. `wiki/log.md` — 최근에 무엇을 했는지 확인
4. 필요하면 `wiki/` 전체에서 관련 키워드 검색

이 과정을 생략하면 중복 페이지를 만들거나, 이미 정리한 내용을 놓치거나, 과거 정리와 충돌할 수 있다.

## 4. 페이지 작성 공통 규칙

### 파일명
- 기본적으로 설명적인 kebab-case를 쓴다.
  - 예: `java-basic-types.md`, `oracle-join.md`, `spring-boot-controller.md`
- 날짜 중심 수업 요약은 `YYYY-MM-DD-topic.md` 형식을 권장한다.
  - 예: `2026-06-29-java-variables.md`
- 한국어 제목은 페이지 내부 `title`에 자연스럽게 쓰고, 파일명은 가능한 한 짧고 안정적인 영문 slug로 둔다.

### YAML frontmatter
`wiki/`의 모든 지식 페이지는 다음 frontmatter를 가진다.

```yaml
---
title: 페이지 제목
created: YYYY-MM-DD
updated: YYYY-MM-DD
type: summary | concept | entity | comparison | query | meta
tags: []
sources: []
status: seed | growing | stable | needs-review
confidence: high | medium | low
---
```

- `sources`에는 참조한 원본 파일 경로를 적는다. 예: `raw/2026-06-29-java.md`
- `status`
  - `seed`: 처음 만든 초안
  - `growing`: 추후 업데이트가 계속 필요
  - `stable`: 여러 자료로 확인되어 비교적 안정적
  - `needs-review`: 사용자의 확인이 필요하거나 모순/불확실성이 있음
- `confidence`
  - `high`: 여러 자료 또는 실습으로 충분히 확인됨
  - `medium`: 한두 자료에 근거하나 일반적으로 타당함
  - `low`: 추정, 불확실, 원본 품질이 낮음

### 링크 규칙
- 관련 페이지가 있으면 반드시 `[[페이지명]]` 또는 `[[경로/페이지명|표시명]]` 형태로 연결한다.
- 새 개념 페이지를 만들 때는 가능한 한 최소 2개 이상의 관련 페이지로 연결한다.
- 아직 페이지가 없지만 앞으로 만들 가치가 있는 개념은 위키링크로 남겨도 된다. 단, `wiki/log.md`나 lint 결과에서 “생성 후보”로 표시한다.

### 출처 규칙
- 원본 자료에 근거한 중요한 주장에는 가능한 한 출처 파일을 명시한다.
- 한 페이지가 여러 원본을 종합하면 문단 끝에 `^[raw/파일명.md]` 형태의 provenance marker를 붙일 수 있다.
- 사용자의 질문에 답하면서 새로 만든 `query` 페이지는 사용한 위키 페이지와 원본 자료를 `sources` 또는 본문에 명시한다.

### 태그 규칙
처음에는 아래 태그를 우선 사용한다. 새 태그가 필요하면 먼저 이 목록을 업데이트한 뒤 사용한다.

- `java`
- `oracle`
- `sql`
- `html`
- `css`
- `javascript`
- `typescript`
- `bootstrap`
- `spring`
- `spring-boot`
- `react`
- `frontend`
- `backend`
- `linux`
- `aws`
- `ci-cd`
- `auth`
- `python`
- `debugging`
- `project`
- `curriculum`
- `study-log`

## 5. 페이지 유형별 작성 규칙

### 5.1 요약 페이지 (`wiki/summaries/`)
수업 하루, 강의 한 단원, 문서 하나, 실습 하나를 요약한다.

권장 구조:

```markdown
# 제목

## 한 줄 요약

## 배운 내용

## 핵심 개념

## 실습 / 예제

## 헷갈린 점 / 질문

## 관련 페이지

## 출처
```

작성 기준:
- 원본 자료의 전체 흐름을 보존하되, 단순 복붙이 아니라 학습자가 다시 읽기 쉬운 형태로 재구성한다.
- 날짜가 명확하면 반드시 제목 또는 frontmatter에 반영한다.
- “오늘 무엇을 배웠는지”를 나중에 추적할 수 있게 커리큘럼 단계와 연결한다.

### 5.2 개념 페이지 (`wiki/concepts/`)
개념, 문법, 패턴, 원리, 오류 해결법 등을 설명한다.

권장 구조:

```markdown
# 개념명

## 정의

## 왜 중요한가

## 핵심 설명

## 예시

## 자주 헷갈리는 점

## 관련 개념

## 출처
```

작성 기준:
- 초보 개발자 관점에서 설명한다.
- 사용자가 어문계열 배경에서 개발자로 전환 중이라는 점을 고려해, 비유와 단계적 설명을 적극 활용한다.
- 단, 틀린 단순화는 피하고 정확한 용어를 함께 제시한다.

### 5.3 엔티티 페이지 (`wiki/entities/`)
기술, 도구, 프레임워크, 라이브러리, 서비스, 명령어 묶음 등을 정리한다.

예: `Java`, `Oracle Database`, `Spring Boot`, `React`, `AWS EC2`, `GitHub Actions`

권장 구조:

```markdown
# 엔티티명

## 무엇인가

## 이 위키에서의 맥락

## 핵심 기능 / 특징

## 관련 개념

## 학습 이력

## 출처
```

### 5.4 비교 페이지 (`wiki/comparisons/`)
두 개 이상의 개념/도구/접근법을 비교한다.

권장 구조:

```markdown
# A vs B

## 비교 목적

## 한눈에 보기

| 항목 | A | B |
|---|---|---|

## 언제 무엇을 쓰는가

## 헷갈리기 쉬운 포인트

## 관련 페이지

## 출처
```

### 5.5 질문 보존 페이지 (`wiki/queries/`)
사용자가 한 질문 중 나중에 다시 볼 가치가 있는 답변을 저장한다.

저장 기준:
- 단순한 용어 확인은 저장하지 않는다.
- 여러 페이지를 종합한 답변, 비교, 로드맵, 오류 해결 과정, 면접 대비 설명처럼 다시 쓸 가치가 있으면 저장한다.

권장 구조:

```markdown
# 질문 제목

## 질문

## 짧은 답

## 자세한 설명

## 참고한 위키 페이지

## 다음에 볼 것
```

## 6. Ingest 작업 순서: 새 원본 자료를 위키에 넣을 때

사용자가 `raw/`에 새 마크다운 파일을 넣고 “처리해줘”, “정리해줘”, “ingest 해줘”라고 하면 다음 순서로 작업한다.

1. **오리엔테이션**
   - `AGENTS.md`, `wiki/index.md`, 최근 `wiki/log.md`를 읽는다.
   - 이미 존재하는 관련 페이지를 찾는다.

2. **원본 읽기**
   - 대상 `raw/` 파일을 읽는다.
   - `raw/` 파일은 절대 수정하지 않는다.
   - 날짜, 수업 주제, 커리큘럼 위치, 핵심 개념, 실습 내용을 파악한다.

3. **요약 페이지 작성**
   - `wiki/summaries/`에 수업/자료 단위 요약 페이지를 만든다.
   - 날짜가 있으면 `YYYY-MM-DD-topic.md` 형식을 우선한다.

4. **개념·엔티티 페이지 업데이트**
   - 원본에서 중심적으로 다룬 개념은 `wiki/concepts/`에 새로 만들거나 기존 페이지를 업데이트한다.
   - Java, Oracle, Spring Boot, React 같은 기술/도구는 `wiki/entities/`에 누적한다.
   - 단순히 한 번 스쳐 지나간 용어는 바로 페이지를 만들지 말고 필요 시 생성 후보로 남긴다.

5. **비교/질문 페이지 생성 여부 판단**
   - 원본이 여러 개념의 차이를 다루거나 사용자가 헷갈려 한 지점이 있으면 `wiki/comparisons/`를 만든다.
   - 대화 중 나온 중요한 설명은 `wiki/queries/`에 보존할지 판단한다.

6. **링크와 출처 정리**
   - 새/수정 페이지에 관련 `[[wikilinks]]`를 추가한다.
   - `sources`와 본문 출처에 원본 파일 경로를 명시한다.

7. **index 업데이트**
   - 새 페이지를 `wiki/index.md`의 알맞은 섹션에 추가한다.
   - 기존 페이지를 크게 바꿨으면 한 줄 설명도 업데이트한다.
   - `Last updated`와 `Total pages`를 갱신한다.

8. **log 업데이트**
   - `wiki/log.md`에 `## [YYYY-MM-DD] ingest | 원본 제목` 형식으로 추가한다.
   - 생성/수정한 파일 목록과 핵심 변경 사항을 적는다.

9. **사용자에게 보고**
   - 생성/수정한 파일을 목록으로 알려준다.
   - 애매하거나 사용자 확인이 필요한 부분을 따로 질문한다.

## 7. Query 작업 순서: 질문에 답할 때

사용자가 학습 내용에 대해 질문하면 다음 순서로 답한다.

1. `wiki/index.md`를 먼저 읽고 관련 페이지 후보를 찾는다.
2. 필요하면 `wiki/log.md`에서 날짜/최근 작업을 확인한다.
3. 관련 `wiki/` 페이지를 읽는다.
4. 위키에 정보가 부족하면 `raw/`에서 관련 원본을 읽는다. 단, `raw/`는 수정하지 않는다.
5. 답변에는 가능한 한 `[[관련 페이지]]`를 언급해 사용자가 Obsidian에서 따라갈 수 있게 한다.
6. 답변이 단순 질의응답을 넘어 비교·정리·로드맵·오류 해결처럼 보존 가치가 있으면 `wiki/queries/` 또는 `wiki/comparisons/`에 파일로 저장할지 판단한다.
7. 보존했다면 `wiki/index.md`와 `wiki/log.md`를 업데이트한다.

## 8. Lint 작업 순서: 위키 점검

사용자가 “위키 점검”, “lint”, “건강검진”, “정리 상태 봐줘”라고 하면 다음을 확인한다.

1. 깨진 위키링크: `[[...]]`가 존재하지 않는 페이지를 가리키는지 확인
2. 고립 페이지: 다른 페이지에서 링크되지 않는 페이지 확인
3. index 누락: `wiki/`에 있는 페이지가 `wiki/index.md`에 빠져 있는지 확인
4. frontmatter 누락/형식 오류 확인
5. 태그가 허용 목록 밖으로 늘어났는지 확인
6. `raw/`를 참조하지 않는 요약 페이지 확인
7. 너무 긴 페이지 확인: 200줄을 넘으면 분할 후보로 표시
8. 오래된 페이지 확인: 새 원본에서 바뀐 내용이 반영되지 않았는지 확인
9. 모순/불확실성 확인: `status: needs-review`, `confidence: low` 페이지를 우선 점검
10. 결과를 `wiki/_meta/`에 lint 리포트로 저장할지 판단하고, `wiki/log.md`에 기록

## 9. 작성 톤과 설명 방식

- 사용자는 개발자로 전환 중이므로, 설명은 초보자에게 친절하되 실제 개발 용어를 피하지 않는다.
- “왜 필요한가 → 어떻게 동작하는가 → 예시는 무엇인가 → 어디서 헷갈리는가” 순서가 좋다.
- 한국어를 기본으로 작성하되, 프로그래밍 용어는 영어 원어를 함께 병기한다.
- 단순 암기보다 연결을 중시한다. 예: `Java 클래스`는 `객체`, `메서드`, `Spring Bean`과 연결한다.
- 학원 진도 추적이 중요하므로 날짜, 수업 주제, 커리큘럼 단계를 놓치지 않는다.

## 10. 변경 원칙

- `raw/`는 읽기 전용이다.
- `wiki/`는 LLM이 책임지고 유지한다.
- 새 페이지를 만들면 반드시 `wiki/index.md`에 등록한다.
- 의미 있는 작업을 하면 반드시 `wiki/log.md`에 남긴다.
- 기존 내용과 충돌하는 새 정보는 덮어쓰기보다 “이전 정리 / 새 정리 / 판단 근거”를 함께 기록한다.
- 대량 변경 전에는 변경 범위를 사용자에게 설명하고, 필요하면 확인을 받는다.
